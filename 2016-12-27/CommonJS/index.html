CommonJS

发表于 2016-12-27

CommonJS 规范涵盖了模块、二进制、Buffer、字符集编码、I/O 流进程环境、文件系统、套接字、单元测试、Web 服务起网关接口、包管理等。

CommonJS模块规范

模块引用

require() 方法接受模块表识，引入一个模块的 API 到当前的上下文环境。

const React = require(‘react’);

模块定义

上下文提供 exports 对象用于导出当前模块的方法和变量，exports 是 module 的属性。在 Node 中一个文件即一个模块，将对象挂载在 exports 上即可导出。

exports.a = function() {}

模块标识

模块标识是传递给 require 符合小驼峰命名的参数，或者是相对 && 绝对路径。

Node 模块的实现

Node 中引入模块的步骤: 路径分析 -> 文件定位 -> 编译执行

  • 核心模块: Node 提供的模块，只执行路径分析，并拥有优先权。
  • 文件模块: 用户编写的模块

require() 对模块的加载采取缓存优先的的方式。

路径分析 && 文件定位

模块标识符分析

Node 模块标识符包含下列几类

  • 核心模块: http、fs、path等: 优先级仅次于缓存加载
  • . 或 .. 开始的相对文件路径 && ／ 开始的绝对文件路径: 将路径转为真实路径 -> 将编译执行后的结果放到缓存中（二次加载）
  • 非路径形式的文件模块: 特殊的文件模块，Node 逐个尝试模块路径中的路径，直到找到目标文件为止，耗时！

文件定位

  • 文件扩展名分析: 标识符不存在的情况，Node 按 .js -> .json -> .node 的次序补足扩展名，依次尝试。尝试过程中调用 fs 模块中模块同步非阻塞式判断文件是否存在。（性能优化：.node,.json 加上扩展名;同步配合缓存;）
  • 目录分析和包: 分析标识符没找到对应文件，但得到一个目录，Node 将目录当作包处理。(package.json -> JSON.parse() -> main: index -> index.(js|json|node) (-> 没找到，抛出异常))

模块编译

定位到具体文件后，Node 会新建一个模块对象，根据路径载入并编译。

  • .js 通过 fs 模块同步读取文件并编译
  • .node C/C++ 编写的扩展文件，通过 dlopen() 方法加载最后编译生成的文件
  • .json 通过 fs 模块同步读取文件,用 JSON.parse() 解析返回结果
  • 其它扩展名被当作 .js 文件引入

JavaScript 模块的编译

在编译的过程中，Node 对文件内容进行了头尾包装，每个文件之间都进行了作用域隔离。包装后的代码通过 vm 原声模块的 runInThisContext() 方法执行，返回一个具体的 function 对象。

执行之后，模块的 exports 属性被返回给调用方，exports 中任何属性和方法都可以被外部调用，但模块中的其余变量和属性则不可以直接被调用。

理想情况下只要赋值给 exports, 但通常情况直接赋值形参会改变行参的引用，但不能改变作用域外的值。

C/C++ 模块的编译

Node 调用 process.dlopen() 方法进行加载和执行。在 Node 架构下 dlopen() 方法在 Windows 和 *nix 下有不同的表现，通过 libuv 兼容层封装。

.node 模块文件不需要编译，它是由 C/C++ 模块之后编译生成的，这里只有加载和执行的过程。执行过程中，模块的 exports 对象与 .node 模块产生联系，返回给调用者。
