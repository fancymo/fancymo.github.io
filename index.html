迭代器是ES6中独立的内建类, 通过实现 [Symbol.iterator]() 和 .next()两个方法你就可以创建自定义迭代器.

举个栗子

Generators－生成器

生成器对象由生成器函数（function*）返回. 同时遵循The iterable protocol 可遍历协议和The iterator protocol 迭代器模式协议.

语法

方法

当一个生成器函数被一个函数体调用时并不会马上执行; 相反, 它会返回一个generator-iterator对象. 每次调用generator-iterator的next()方法将会执行这个函数体至下一个yield表达式并且返回一个结果. 当函数执行完或者执行到一个return时抛出一个StopIteration exception异常.

与普通函数区别:

  • 普通函数使用 function 声明，而生成器函数使用 function* 声明;
  • 在生成器函数内部, 有一种类似 return 的语法: 关键字 yield , 在生成器的执行过程中, 遇到yield表达式立即暂停, 后续可恢复执行状态.

内部原理：每当生成器执行yields语句，生成器的堆栈结构（本地变量、参数、临时值、生成器内部当前的执行位置）被移出堆栈.然而，生成器对象保留了对这个堆栈结构的引用（备份），所以稍后调用.next()可以重新激活堆栈结构并且继续执行.

this 的理解

全局上下文

在全局运行上下文中(在任何函数体外部), this 指代全局对象.

函数上下文

在函数内部, this 的值取决于函数是如何调用的.

直接调用

非严格模式下, this 的值总是一个对象且默认为全局对象.

在严格模式下, this 是在进入运行环境时设置的. 若没有定义, this的值将维持undefined状态。同时它也能设置成任意值.

对象方法中的 this

this 是调用该函数的对象, this 的绑定只受最靠近的成员引用的影响.

getter 与 setter 中的 this

作为 getter 或 setter 函数都会绑定 this 到从设置属性或得到属性的那个对象

构造函数中的 this

当一个函数被作为一个构造函数来使用(使用 new 关键字), 它的 this 与即将被创建的新对象绑定.

call 和 apply

当一个函数的函数体中使用了 this 关键字时, 通过所有函数都从 Function 对象的原型中继承的 call() 方法和 apply() 方法调用时, 它的值可以绑定到一个指定的对象上.

bind 方法

调用 f.bind(someObject) 会创建一个与f具有相同函数体和作用域的函数, 但是在这个新函数中, this将永久地被绑定到了bind的第一个参数.

DOM事件处理函数中的 this

当函数被用作事件处理函数时，它的this指向触发事件的元素

内联事件处理函数中的 this

当代码被内联处理函数调用时，它的this指向监听器所在的DOM元素

《深入浅出Node.js》(一) CommonJS

发表于 2016-12-27

CommonJS 规范涵盖了模块、二进制、Buffer、字符集编码、I/O 流进程环境、文件系统、套接字、单元测试、Web 服务起网关接口、包管理等。

1. CommonJS模块规范

1.1模块引用

require() 方法接受模块表识，引入一个模块的 API 到当前的上下文环境。

const React = require(‘react’);

1.2 模块定义

上下文提供 exports 对象用于导出当前模块的方法和变量，exports 是 module 的属性。在 Node 中一个文件即一个模块，将对象挂载在 exports 上即可导出。

exports.a = function() {}

1.3 模块标识

模块标识是传递给 require 符合小驼峰命名的参数，或者是相对 && 绝对路径。

2. Node 模块的实现

Node 中引入模块的步骤: 路径分析 -> 文件定位 -> 编译执行

  • 核心模块: Node 提供的模块，只执行路径分析，并拥有优先权。
  • 文件模块: 用户编写的模块

require() 对模块的加载采取缓存优先的的方式。

2.1 路径分析 && 文件定位

2.1.1 模块标识符分析

Node 模块标识符包含下列几类

  • 核心模块: http、fs、path等: 优先级仅次于缓存加载
  • . 或 .. 开始的相对文件路径 && ／ 开始的绝对文件路径: 将路径转为真实路径 -> 将编译执行后的结果放到缓存中（二次加载）
  • 非路径形式的文件模块: 特殊的文件模块，Node 逐个尝试模块路径中的路径，直到找到目标文件为止，耗时！

2.1.2 文件定位

  • 文件扩展名分析: 标识符不存在的情况，Node 按 .js -> .json -> .node 的次序补足扩展名，依次尝试。尝试过程中调用 fs 模块中模块同步非阻塞式判断文件是否存在。（性能优化：.node,.json 加上扩展名;同步配合缓存;）
  • 目录分析和包: 分析标识符没找到对应文件，但得到一个目录，Node 将目录当作包处理。(package.json -> JSON.parse() -> main: index -> index.(js|json|node) (-> 没找到，抛出异常))

2.2 模块编译

定位到具体文件后，Node 会新建一个模块对象，根据路径载入并编译。

  • .js 通过 fs 模块同步读取文件并编译
  • .node C/C++ 编写的扩展文件，通过 dlopen() 方法加载最后编译生成的文件
  • .json 通过 fs 模块同步读取文件,用 JSON.parse() 解析返回结果
  • 其它扩展名被当作 .js 文件引入

2.2.1 JavaScript 模块的编译

在编译的过程中，Node 对文件内容进行了头尾包装，每个文件之间都进行了作用域隔离。包装后的代码通过 vm 原声模块的 runInThisContext() 方法执行，返回一个具体的 function 对象。

执行之后，模块的 exports 属性被返回给调用方，exports 中任何属性和方法都可以被外部调用，但模块中的其余变量和属性则不可以直接被调用。

理想情况下只要赋值给 exports, 但通常情况直接赋值形参会改变行参的引用，但不能改变作用域外的值。

2.2.2 C/C++ 模块的编译

Node 调用 process.dlopen() 方法进行加载和执行。在 Node 架构下 dlopen() 方法在 Windows 和 *nix 下有不同的表现，通过 libuv 兼容层封装。

.node 模块文件不需要编译，它是由 C/C++ 模块之后编译生成的，这里只有加载和执行的过程。执行过程中，模块的 exports 对象与 .node 模块产生联系，返回给调用者。

2.2.3 JSON 文件的编译

3. 核心模块

Node 目录下src － C/C++ , lib - JavaScript.

3.1 JavaScript 核心模块的编译

  1. Node 采用 V8 的 js2c.py 工具，将 JS 转存 C/C++ 数组。

    在这个过程中，将 JavaScript 代码以字符串的形式存在 node 命名空间中，不可直接执行。启动 Node 进程时，将 JavaScript 代码加载进内存。

  2. 编译
    lib 下所有文件没有定义require、 module、 exports变量，先进行头尾包装，执行导出。

3.2 C/C++ 核心模块编译

核心模块中，一种是全部由 C/C++ 编写，还有一种由 C/C++ 完成核心部分，JavaScript主外实现封装。由纯 C/C++ 编写的统称为内建模块，不由外部文件直接调用。执行时被编译进二进制文件中 直接执行。

包描述文件用于描述包的相关信息。

- name 包名

export 和 module.exports 的区别

发表于 2016-12-26

Nodejs – exports

Nodejs 中使用 exports 对象创建函数

ES6 – export

export 语句用于从给定文件（或模块）导出函数和对象。导出类型可以为 function class var let const.

  • 命名导出

export { myFunction }; // 导出一个函数声明
export const foo = Math.sqrt(2); // 导出一个常量

  • 默认导出 (每个脚本只能有一个)

export default myFunctionOrClass // 或者 ‘export default class {}’

  • 重命名

export { variable1 as alias };

import && export – depth

JS 引擎运行一个模块，将依次执行下列四步：

  1. 解析－Parsing: 读取源码，检查语法错误；
  2. 加载－Loading: 加载所有 import 进来的包；
  3. 链接－Linking: 为每个加载进来的模块创建模块作用块，并填充，包括它们从别的包 import 进去的模块；
  4. 编译－Run time: import 进程结束，

Static vs. dynamic:

dynamic – JavaScript

  1. import and export are allowed only at toplevel in a module.
  2. All exported identifiers must be explicitly exported by name in the source code.
  3. Module objects are frozen. There is no way to hack a new feature into a module object, polyfill style.
  4. All of a module’s dependencies must be loaded, parsed, and linked eagerly, before any module code runs. There’s no syntax for an import that can be loaded lazily, on demand.
  5. There is no error recovery for import errors. An app may have hundreds of modules in it, and if anything fails to load or link, nothing runs. You can’t import in a try/catch block.
  6. There is no hook allowing a module to run some code before its dependencies load. This means that modules have no control over how their dependencies are loaded.

exports 导出传统的模块实例，且当 module.exports 存在属性时，exports 无效
export 导出的特别对象类型，用 require 可以获取(可能会封装层 –_esmodule)，import 为 undefine
